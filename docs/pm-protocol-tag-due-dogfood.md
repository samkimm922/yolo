## 需求案例

目标：团队成员可以创建和管理待办标签、按标签筛选、设置和修改到期时间，并在到期前收到站内提醒。项目中预先存在四个对应的领域文件：

- `src/domain/tag.ts`
- `src/features/todo-filter.ts`
- `src/domain/todo.ts`
- `src/services/reminder-scheduler.ts`

## 交互体验对比

| 对比项 | 旧基线（用户提供） | 本次实测 |
| --- | --- | --- |
| 开始方式 | 直接进入扁平的 10+1 题 | 先问现有办法、不做后果、最小版本，再明确判断“继续/不继续” |
| 提问组织 | 题库顺序，层次不明确 | 角色/现状/痛点 → 一天的场景 → 例外/边界 → 验收证据 |
| 中途纠偏 | 全部问完后一次性 playback | 初始场景确认 + 每层确认门；修改已确认答案会使当前层及后续确认失效 |
| 复述 | 回显原回答 | 合成为“谁在什么时候进入流程、当前怎么做、遇到什么问题、期望看到什么”的一天场景 |
| 认知负担 | 开放题无落点 | 当前问题携带基于已知上下文的推荐/示例答案，可直接确认或修改 |
| 需求交接 | 没有唯一的结构化清单 | 在批准前回放并确认 R-001～R-004，PRD 只从这四项业务能力生成任务 |

本次非 UI 流程共 19 个问题节点，其中包括前提判断、五个内容哈希绑定的确认门（四层 + R 清单）和最终批准。CLI 还验证了显式传入未来问题 ID 不能绕过当前确认门。

## PRD 质量对比

| 指标 | 旧基线（用户提供） | 本次实测 |
| --- | --- | --- |
| 业务任务数 | 3 个近似任务 | 4 个领域任务 |
| 任务标题 | 雷同，无法看出领域边界 | 标签管理、标签筛选、到期时间、提醒调度各自唯一 |
| 文件落点 | 同一个编造的占位文件 | 四个任务分别落到上述四个真实文件 |
| 假文件 | 有 | 0；测试同时拒绝 `feature-<hash>`/placeholder 命名 |
| 原始访谈正文 | 约占 PRD 60% | 不进入 PRD；PRD 只保存 demand 产物引用、问题 ID 和计数摘要 |
| demand 区块占比 | 约 60% | 23,413 / 89,372 字节，约 26.2% |

“设置和修改到期时间”被识别为同一字段生命周期，而不是两个无上下文的故事碎片；同时，“标签模型”“筛选逻辑”“到期字段”“通知调度”保持为四个独立领域任务。
